# CLI

`packages/cli` produces the `dreamer` binary that users install via
`npx dreamer` or a standalone compiled binary. It bundles:

- The built web UI (embedded via Bun's `with { type: "file" }` import).
- An Elysia API server spawned as a child process.
- A headless REPL + one-shot agent invocation mode.
- arduino-cli resolution + auto-install plumbing (shared with the API).

## Entry points

- `packages/cli/src/index.ts` — CLI arg dispatch. Handles `headed`, `run`,
  `compile`, `flash`, `list-ports`, `config get|set|list|path`, `logs`,
  `crashes`, `self-update`, and a default one-shot agent mode.
- `packages/cli/src/headed.ts` — the GUI launcher. Spawns the API, starts
  the static web UI server, opens the browser.
- `packages/cli/src/repl.ts` — interactive REPL over the agent.
- `packages/cli/src/runner.ts` — single-shot `runAgent(prompt)` wrapper.

Most CLI surface is documented in `HELP_TEXT` from `cli-args.ts`.

## Static web UI server

File: `packages/cli/src/web-ui.ts`

`startStaticWebUI(port, apiOrigin, appOrigin)` spins up a `Bun.serve()` that
reads asset paths from `EMBEDDED_ASSETS` (the generated manifest, see
below) and serves them with correct MIME types + cache headers.

Two non-obvious bits:

1. **Runtime config injection.** `/index.html` gets a `<script>` prepended
   to `<head>` that sets `window.__DREAMER__ = { apiOrigin, appOrigin,
   preferAvr: true }`. The frontend's `@dreamer/config` reads this before
   the Vite bundle loads, so the React app knows the actual API port
   (picked at runtime, since the user may not have 4111 free).
2. **`preferAvr: true`** is forced in CLI mode because the shipped binary
   guarantees arduino-cli is available, so real compilation always works
   and the frontend can skip its transpile fallback.

SPA fallback: any path with no extension that isn't in the manifest falls
back to `/index.html`. Extensioned paths that miss return 404. Hashed
asset filenames (Vite's `-Abc12345.js` pattern) get
`max-age=31536000, immutable`; everything else gets `max-age=0,
must-revalidate`.

## The asset manifest

File: `packages/cli/src/web-ui-manifest.generated.ts`

Auto-generated by `scripts/generate-asset-manifest.ts` (at the repo root)
as part of `bun run build:webui`:

```ts
// Each line imports a static file path with `with { type: "file" }` so Bun
// knows to preserve the file at bundle time.
import asset__index_html from "../../app/dist/index.html" with { type: "file" }
// ... (one per file under packages/app/dist/)

export const EMBEDDED_ASSETS: Record<string, string> = {
  "/index.html": asset__index_html,
  "/": asset__index_html,
  "/assets/index-Abc12345.js": asset__index_js,
  // ...
}
```

The `with { type: "file" }` attribute is Bun-specific — it tells the
compiler to embed the file contents in the binary and hand the running
process an absolute path to the extracted asset. At runtime,
`Bun.file(EMBEDDED_ASSETS[path])` streams the bytes.

The manifest is regenerated every build, so the hashed filenames change.
**Do not edit by hand** — it has `@ts-nocheck` and is overwritten on each
`bun run build:cli`.

## Hosted mode vs. CLI mode

There are two independent static-UI paths for different deploy targets:

| Mode | Static route source | Served from | Activation |
| --- | --- | --- | --- |
| CLI binary | `packages/cli/src/web-ui.ts` | Binary-embedded manifest | `dreamer headed` |
| Hosted (e.g. Railway) | `packages/api/src/routes/web-ui-static.ts` | `packages/app/dist/` on disk | `DREAMER_HOSTED=1` + dist present |

The hosted route mounts as an Elysia plugin (same port as the API) and
uses the root app's `onError({ code: "NOT_FOUND" }, ...)` hook for SPA
fallback. Plugin-level wildcard matchers were unreliable for this —
`routes/web-ui-static.ts:79` comments on the reason.

### Opt-outs

- `DREAMER_API_ONLY=1` — disables the static route even when `dist/` is
  present. Used when a single Docker image backs two services (a frontend
  service that serves UI, and an API-only service that must not).
  Implemented at `routes/web-ui-static.ts:86`.
- `DREAMER_HOSTED=1` — flips CORS to `origin: true` and sets a log flag.
  Not strictly required for the static route to work (existence of
  `dist/` is the gating condition) but used by `index.ts` to tell the
  user about it in the banner.

## arduino-cli toolchain

File: `packages/api/src/toolchain.ts` (shared — both the API server and the
CLI use it).

Resolution order for the arduino-cli binary:

1. `$DREAMER_ARDUINO_CLI` — explicit override. If set but the file doesn't
   exist, throws `ArduinoCliMissingError`.
2. `$DREAMER_HOME/bin/arduino-cli` — managed install location.
3. `PATH` lookup.
4. If `$DREAMER_AUTO_INSTALL=1`, fetch the arduino-cli installer into
   `$DREAMER_HOME/bin` and retry.

Cores (e.g. `arduino:avr`) are ensured on first compile via
`ensureArduinoCliCore(fqbn)`, which runs `arduino-cli core install
arduino:avr` if the core is missing. Core installation is gated by the
same `DREAMER_AUTO_INSTALL` flag when in non-interactive CLI mode.

## Crash reporting + telemetry

- `crash-reporter.ts` installs a `process.on('uncaughtException')` handler
  at boot (`index.ts:33`). Crashes land in `$DREAMER_HOME/crashes/*.json`.
  `dreamer crashes list|show|clear` surfaces them.
- `telemetry.ts` is opt-in (`dreamer config set telemetry true`). Emits
  anonymized events to a configurable endpoint. Off by default; no PII.

## Self-update

`self-update.ts` fetches the latest release from the configured channel
(`stable` or `beta`) and replaces the binary atomically. Channel is stored
in `$DREAMER_HOME/config.json`. See `paths.ts:6` for the full config path
resolution.

## Config

`config.ts`:

- `$DREAMER_HOME/config.json` — persistent config (`anthropic.apiKey`,
  `telemetry.enabled`, `updates.channel`).
- `ensureApiKey()` — checked at CLI boot for commands that need the
  agent. Throws `ApiKeyMissingError` (exit code 2) if missing.

Getter/setter for the key goes through `setApiKey` / `clearApiKey` so the
file's JSON shape is stable.

## Build pipeline

```
bun run build:cli
├─ bun run build:webui
│  ├─ bun run --cwd packages/app build    # Vite production build → packages/app/dist/
│  └─ bun run scripts/generate-asset-manifest.ts
│     └─ overwrites packages/cli/src/web-ui-manifest.generated.ts
└─ bun run --cwd packages/cli build:all
   └─ bun build --compile  (with the new manifest baked in)
```

Running any CLI command requires the manifest to be in sync with the last
frontend build — the typechecker won't catch a stale manifest (the paths
are strings).
